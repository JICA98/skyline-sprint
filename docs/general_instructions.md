# General Instructions — Skyline Sprint PS4 Homebrew

## 1. Mission

Create a complete, legally redistributable, open-source PlayStation 4 homebrew game named **Skyline Sprint** using the OpenOrbis PS4 Toolchain:

- Repository: https://github.com/OpenOrbis/OpenOrbis-PS4-Toolchain
- Preferred baseline when this specification was written: **v0.5.4**
- Language: C++17 where supported by the current OpenOrbis samples
- Rendering and basic platform services: use the currently working SDL2/OpenOrbis sample implementation unless repository inspection proves another official sample is more reliable
- Target output: installable PS4 homebrew `.pkg`, corresponding `eboot.bin`, complete source code, reproducible build scripts, checksums, and documentation
- Primary use: a small but meaningful compatibility test for the Linsui/shadPS4 Android runtime
- Secondary use: a playable, original infinite side-scrolling platform game

The game must be inspired only by the broad platform-game genre. It must not copy Mario characters, level layouts, names, sounds, sprites, music, visual identity, or other protected content.

## 2. Required Outcome

The final repository must allow a developer to:

1. Clone the project.
2. Install or point to a supported OpenOrbis release.
3. Run one documented build command.
4. Produce an installable PS4 homebrew package.
5. Run a desktop SDL2 development build when desktop dependencies are available.
6. Verify the output using documented checksums.
7. Redistribute the source, original assets, and homebrew package legally.
8. Launch the package in Linsui/shadPS4 and obtain useful logs and on-screen diagnostics.

Do not stop after scaffolding. Do not leave the project as a prototype that merely opens a window. The final build must include a complete game loop, procedural level generation, player movement, collision, collectibles, failure/restart flow, menus, diagnostics, and packaging.

## 3. Authority and Source-of-Truth Rules

Before editing code:

1. Read these files completely:
   - `general_instructions.md`
   - `ui_ux_design.md`
   - `tasks.md`
2. Inspect the current OpenOrbis repository and release being used.
3. Inspect:
   - the SDL2 mini-game/sample;
   - the input/pad sample;
   - package metadata from at least one working sample;
   - the current build scripts and Makefiles;
   - any relevant documentation in `docs/`;
   - the release notes for the selected OpenOrbis version.
4. Record the selected OpenOrbis commit or release tag in the project README and build metadata.
5. Prefer, in descending order:
   - code that compiles in the current official samples;

   - current official documentation;
   - older examples only when verified against the selected release.

Never invent an OpenOrbis function, library name, linker flag, package command, controller constant, or module dependency. Search the installed toolchain and official samples, then use the exact verified form.

If a documented instruction conflicts with a compiling current sample, follow the current sample and document the discrepancy.

The entire project must be safe to publish publicly and to attach to an emulator compatibility report.

Use a project licence compatible with all included code. GPL-3.0-or-later is acceptable for the project when that is compatible with the reused OpenOrbis sample code. Add:

## 5. Product Definition

### 5.1 Game concept

**Skyline Sprint** is an endless 2D platform runner set above a stylized geometric city.

The player controls **Pulse**, an original small luminous courier represented by a rounded geometric character. The world scrolls from right to left as Pulse runs across rooftops, platforms, bridges, and floating energy rails.

Core actions:

- Move left and right within a limited camera zone.
- Jump.
- Optional short second jump if enabled by the final tuning.
- Drop faster while airborne.
- Pause.
- Restart immediately after failure.

Core objectives:

- Stay alive.
- Collect energy shards.
- Travel as far as possible.
- Build a score multiplier through consecutive pickups and clean landings.

Hazards:

- Gaps.
- Static barriers.
- Moving low-risk drones represented by original geometric shapes.
- Collapsing platforms after the base game is stable.

The first shippable version must remain small. Do not add combat, online play, complex inventory, story scenes, or a large asset pipeline.

### 5.2 Compatibility-test features

The game must function as a useful emulator test package. Include:

- Build identifier on the title screen and diagnostics overlay.
- OpenOrbis release/commit identifier.
- Current procedural seed.
- Frame counter.
- Average, minimum, and maximum frame time over a rolling window.
- Simulation tick count.
- Rendered object count.
- Active chunk count.
- Controller connection state.
- Audio initialization state.
- Last nonfatal warning.
- A small in-memory circular log displayed on a diagnostics screen.
- File logging when a writable path is available through verified OpenOrbis APIs.
- Deterministic fixed-seed mode.
- Stress mode that increases decorative objects and particles without changing gameplay correctness.
- Input visualizer showing current controller buttons and analog values.
- Automatic attract/test mode that can play or simulate a repeatable route and automatically restart after failure.

Diagnostics must be available without external tools. Normal gameplay should not be visually cluttered.

Adapt this layout when the verified OpenOrbis sample build system requires it, but preserve separation between:

- deterministic gameplay simulation;

### 6.1 Platform abstraction

Create a minimal platform interface for:

- initialization and shutdown;
- high-resolution monotonic time;
- controller polling;
- video/render initialization;
- audio initialization;
- file paths and logging;
- application exit requests.

Implement:

- `platform_ps4`;
- `platform_desktop`.

Do not force PS4-only headers into deterministic gameplay modules.

### 6.2 Game loop

Use a fixed simulation timestep:

- simulation target: 60 ticks per second;
- accumulator-based update loop;
- maximum frame delta clamp to prevent a debugger stall from generating a huge physics step;
- optional render interpolation;
- no gameplay decisions based on rendering frame rate.

Recommended behavior:

```text
poll input
measure elapsed time
clamp elapsed time
accumulate
while accumulator >= fixed_step:
    update deterministic simulation
    accumulator -= fixed_step
render interpolated state
present
```

Avoid uncontrolled busy loops. Use the exact wait/present behavior demonstrated by the verified sample.

### 6.3 Determinism

Procedural generation and automatic test mode must be repeatable.

- Use an explicitly implemented PRNG such as PCG32 or xorshift with a documented algorithm.
- Never use platform `rand()` for level generation.
- Store the seed in game state.
- Generate chunks solely from seed, chunk index, and difficulty tier.
- Keep core simulation free of wall-clock randomness.
- Display the seed after failure so testers can reproduce a problem.
- Add deterministic unit tests on desktop.

### 6.4 Infinite-world strategy

Use chunk-based generation rather than an ever-growing level.

- Divide the world into fixed-width chunks.
- Keep only chunks around the camera.
- Generate ahead of the player.
- Retire chunks safely after they leave the camera.
- Reuse object storage where practical.
- Keep world coordinates bounded by periodically rebasing the origin or using chunk-local coordinates.
- Ensure platform patterns always have at least one reachable route.

Generation rules must have explicit difficulty limits for:

- maximum horizontal gap;
- maximum vertical step;
- moving-platform speed;
- obstacle density;
- minimum landing width;
- recovery platform frequency.

Create a validation pass for every generated chunk. If validation fails, regenerate with a deterministic fallback pattern.

### 6.5 Physics

Use a simple AABB or capsule-like controller rather than a full physics engine.

Required behavior:

- horizontal acceleration and deceleration;
- maximum run speed;
- gravity;
- jump impulse;
- coyote time;
- jump input buffering;
- variable jump height when the jump button is released early;
- swept or sub-stepped collision sufficient to prevent tunneling;
- stable landing;
- one-way platforms only if they can be implemented reliably;
- kill plane below the world;
- immediate and deterministic restart.

Physics constants must be centralized in one tuning file or structure.

### 6.6 Rendering

Prefer the rendering approach used by the selected OpenOrbis SDL2 sample.

- Target a logical 1920×1080 coordinate space.
- Preserve aspect ratio.
- Keep essential UI inside a 5% safe-area margin.
- Use nearest or linear filtering consistently with the geometric art style.
- Batch or minimize state changes where possible.
- Avoid unsupported shaders and advanced GPU features.
- Prefer simple textures, rectangles, lines, and a small particle pool.
- Make all resources load from package-relative paths verified from a working sample.
- Present a readable on-screen fatal-error screen if initialization fails after video is available.

### 6.7 Input

Use the exact verified controller path from the current SDL2 or pad sample.

Default control intent:

- Left stick or D-pad: horizontal movement.
- Cross: jump/confirm.
- Circle: back.
- Options: pause.
- Triangle: toggle automatic test mode from the title or diagnostics screen.
- Square: toggle stress mode from diagnostics.
- L3: toggle compact diagnostics overlay.
- R3: open full diagnostics screen.
- L1/R1: change diagnostics pages.
- Touchpad press, only when verified and reliable: input visualizer shortcut.

Do not rely on controller button constants until verified from the selected SDK headers/sample.

Handle controller disconnection gracefully and display a reconnect message.

### 6.8 Audio

Audio is desirable but must not block the build.

- Generate small original PCM WAV effects for jump, pickup, landing, menu movement, and failure.
- Use a verified SDL/OpenOrbis audio path.
- No copyrighted music.
- A simple generated ambient loop is optional.
- If audio initialization fails, continue in silent mode and record the failure in diagnostics.
- Provide a settings toggle for mute.
- Cap simultaneous sound instances.

### 6.9 Text

To avoid uncertain external font dependencies, prefer one of:

1. a tiny original 5×7 or 8×8 bitmap font stored in source; or
2. the current official OpenOrbis font sample plus a bundled permissive font and complete licence.

The font must cover all characters used by the game and diagnostics.

## 7. Build and Packaging Rules

### 7.1 Toolchain acquisition

The OpenOrbis README recommends using a release archive/installer when prebuilt binaries are needed. A plain source clone may require rebuilding libraries and tools.

The agent must:

- verify the current recommended installation path;
- pin the release or commit used;
- verify `clang` and `lld`;
- verify `OO_PS4_TOOLCHAIN`;
- verify all tools used by the current sample package flow;
- record tool versions in `BUILDING.md`;
- never download or use the proprietary Sony SDK.

### 7.2 Build commands

Provide wrappers that fail fast:

```bash
./build_desktop.sh
./build_ps4.sh
./clean.sh
./scripts/make_release.sh
```

`build_ps4.sh` must:

1. validate required environment variables;
2. validate required tool executables and libraries;
3. clean or use isolated build directories;
4. compile with warnings enabled;
5. create the PS4 executable using the current OpenOrbis process;
6. construct package metadata from a current working sample;
7. create the package;
8. copy outputs to `dist/`;
9. print file sizes and SHA-256 checksums.

Do not silently continue after a failed command.

### 7.3 Package identity

- Create a unique homebrew package identity.
- Never reuse a retail game title ID.
- Derive the valid ID format and required metadata fields from the selected current OpenOrbis sample.
- Use the game name `Skyline Sprint`.
- Use version `01.00` for the first release unless current tooling requires another format.
- Include a simple original icon and splash image.
- Store package metadata in source control.
- Document any system modules copied from the OpenOrbis release and why they are needed.

### 7.4 Reproducibility

- Avoid timestamps embedded in generated game data where possible.
- Generate assets deterministically.
- Store the selected toolchain tag/commit.
- Store SHA-256 values for release artifacts.
- Add a clean-build verification script.
- Make the release script refuse uncommitted generated assets unless explicitly overridden.
- Include exact commands used to build the release.

## 8. Quality Rules

### 8.1 Compiler and code quality

- Enable useful warnings supported by the selected compiler.
- Treat new project warnings as errors where practical.
- Avoid exceptions and RTTI if the verified target setup does not support them reliably.
- Avoid unbounded allocations during gameplay.
- Avoid per-frame file I/O.
- Avoid large standard-library features unless proven to work in the selected sample/toolchain.
- Prefer fixed-capacity containers in hot paths.
- Ensure every initialization has a corresponding shutdown.
- Validate all asset reads.
- Check every platform API return value.
- Keep functions focused and files reasonably small.

### 8.2 Performance budgets

Normal mode target:

- 60 simulation ticks per second.
- 60 rendered frames per second where the host permits.
- Fewer than 500 simultaneously rendered world objects.
- Fixed particle pool.
- No continual heap growth.
- Stable memory use after five minutes.
- No chunk accumulation.
- No log spam in normal mode.

Stress mode may raise object counts but must remain bounded.

### 8.3 Failure behavior

On failure:

- Never show a blank screen when a readable diagnostic can be shown.
- Log the failed subsystem and numeric result code.
- Fall back where safe, especially for audio.
- Do not continue after critical renderer/package-resource failure.
- Preserve the last 128 log messages in memory.
- Show the current seed and build ID on gameplay crashes detectable by the application.

## 9. Testing Strategy

### 9.1 Desktop tests

Add host-side tests for:

- PRNG known vectors;
- same seed producing identical chunks;
- chunk reachability constraints;
- collision landing;
- coyote time;
- jump buffering;
- death/restart state reset;
- world-origin rebasing;
- fixed timestep consistency across different render delta sequences;
- bounded active chunks and objects after simulated long runs.

### 9.2 Manual desktop test

Run:

- title flow;
- keyboard/controller input;
- normal play for at least ten minutes;
- repeated restart;
- pause/resume;
- seed replay;
- automatic test mode;
- stress mode;
- mute/unmute;
- resize/aspect-ratio behavior.

### 9.3 PS4 or emulator test

Record:

- host/hardware or emulator version;
- package checksum;
- toolchain tag/commit;
- firmware/runtime information supplied by the tester;
- startup result;
- video result;
- input result;
- audio result;
- ten-minute stability;
- average/min/max frame time;
- any log or screenshot.

Do not claim real-PS4 success unless a tester actually verified it.

## 12. Definition of Done

The project is done only when all of the following are true:

- The desktop build compiles and is playable.
- The deterministic tests pass.
- The PS4 target compiles with the pinned OpenOrbis version.
- `eboot.bin` is produced.
- An installable package is produced by the verified package workflow.
- The package contains only redistributable material.
- Title, pause, gameplay, failure, diagnostics, stress, input visualizer, and automatic-test modes work.
- Infinite generation remains bounded during a long simulation.
- The release directory contains source/version metadata and SHA-256 checksums.
- Build instructions have been tested from a clean directory.
- Remaining unverified target behavior is explicitly documented.
- No placeholder assets, TODO-only systems, fake success claims, or proprietary files remain.
